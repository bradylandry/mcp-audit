## Medium

### False positive: `mcp.run()` flagged as subprocess call
- **File**: `audit/ast_scan.py:375`
- **Issue**: The subprocess detection check has two conditions:
  ```python
  if name in _SUBPROCESS_FUNCS or any(
      name.endswith("." + sp.split(".")[-1])
      for sp in _SUBPROCESS_FUNCS
      if sp.startswith("subprocess")
  ):
  ```
  The second arm matches any method call whose name ends with `.run`, `.call`, `.check_call`, `.check_output`, `.Popen`, `.getoutput`, or `.getstatusoutput`. This is far too broad: `mcp.run()`, `app.run()`, `server.run()`, `scheduler.run()` â€” all common Python patterns â€” match and are flagged as subprocess calls. Every FastMCP server calls `mcp.run()` at the end of the file, so every FastMCP-based MCP server being audited will receive a spurious MCP001 subprocess finding worth âˆ’2 points. This directly contradicts the tool's stated contract ("findings are only real concerns").
- **Fix**: Remove the second arm of the condition entirely. The `_SUBPROCESS_FUNCS` catalog already contains the canonical fully-qualified names (`subprocess.run`, `os.system`, etc.). The heuristic was presumably intended to catch `subprocess.run(...)` called as a bound method, but that pattern is uncommon and the false-positive rate from `.run()` suffix matching is unacceptable. If catching `runner.run(...)` when `runner` is a `subprocess.Popen` instance is desired, add a receiver-name heuristic similar to `_is_http_client_call` that requires the receiver to look like a subprocess handle.

### Dead ternary: `sev` always `"high"` in `_check_string_content`
- **File**: `audit/findings.py:338`
- **Issue**:
  ```python
  sev: Severity = "high" if bidi_count > 0 else "high"
  ```
  Both branches return `"high"`. The preceding comment says "Plain zero-width chars are usually a hidden-content attack vector but lower-severity than bidi inversion," making clear the intent was to use a different severity for plain zero-width chars (probably `"medium"`). The `deduct` variable correctly differentiates (3 for bidi, 2 for plain), but the severity label is stuck at `"high"` for both, meaning `--max-severity medium` CI gating won't work as documented for plain zero-width findings.
- **Fix**: `sev: Severity = "high" if bidi_count > 0 else "medium"`. The deduction values are already correct; this is a one-word fix.

## Low

### `_check_dependencies` only fires when ALL deps use loose pins
- **File**: `audit/findings.py:428`
- **Issue**:
  ```python
  if loose and len(loose) == len(s.dependencies):
  ```
  The finding is only emitted if every declared dependency is unpinned. A package with 4 loose deps and 1 pinned dep produces no finding. The intent (flagging loose deps) isn't served by this guard.
- **Fix**: Remove the `len(loose) == len(s.dependencies)` condition. Emit the finding whenever there are any loose deps, listing which ones are loose. Adjust the title from "All N dep(s) use loose version pins" to "N of M dep(s) use loose version pins".

### Python 3.10 TOML fallback silently returns empty deps
- **File**: `audit/ast_scan.py:577`
- **Issue**: On Python 3.10, `_parse_deps` returns `([], "pyproject.toml (3.11+ required for full parse)")`. The `dep_source` string contains the explanation, but `dependencies` is empty â€” meaning the dependency dimension check is silently skipped. A caller on 3.10 auditing a package with loose pins sees no dep findings and has no obvious signal that the check was skipped.
- **Fix**: Emit a low-severity info finding when `dep_source` contains the `"3.11+ required"` sentinel, so the report says "Dependency analysis unavailable (Python 3.10)". Alternatively, add `tomli` as a conditional dep for 3.10 and note it in the docs.
