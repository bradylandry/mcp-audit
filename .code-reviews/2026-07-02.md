## Medium

### `--max-severity` silently ignored when `--json` is passed
- **File**: `audit/cli.py:188-191`
- **Issue**: The `--json` branch returns `0` unconditionally before `--max-severity` is evaluated. `--json` and `--max-severity` are in separate argument groups so they can be combined, but doing so (`mcp-audit /path --json --max-severity high`) always exits 0, silently dropping the CI severity gate. The `--json-report` and default-markdown paths correctly check `args.max_severity` before returning; `--json` was missed.
- **Fix**: Before the `return 0` at line 191, add `if args.max_severity: return _exit_code_from_max_severity(report, args.max_severity)`. This requires running `build_report(scan_result)` before the `--json` early-return, or passing the scan result through — see the double-scan finding below for the natural fix opportunity.

### Double scan on `--json` code path
- **File**: `audit/cli.py:186-191`
- **Issue**: `scan()` is called unconditionally at line 186, producing `scan_result`. If `--json` is set, `scan_to_json()` is called at line 189, which internally calls `scan()` again and throws away `scan_result`. On a large target with many `.py` files, this doubles scan time silently. The first call's result is never used.
- **Fix**: Restructure so `--json` serializes the already-computed `scan_result` rather than re-scanning. `scan_to_json` could accept an optional pre-computed `ScanResult`, or the CLI could inline the JSON serialization using the existing `_call_dict` helpers. Resolving this also enables the `--max-severity` fix above.

### Dead severity branch in zero-width-unicode check
- **File**: `audit/findings.py:338`
- **Issue**: `sev: Severity = "high" if bidi_count > 0 else "high"` — both branches evaluate to `"high"`, making the conditional dead. The adjacent comment explicitly states "Plain zero-width chars are usually a hidden-content attack vector but lower-severity than bidi inversion," implying non-bidi zero-width chars should be `"medium"`. Because the severity is always `"high"`, `--max-severity medium` will catch zero-width-space findings even for the intended-lower-severity case, and callers reading `severity` from `--json-report` get no signal on which type was found.
- **Fix**: Change the `else` branch to `"medium"`: `sev: Severity = "high" if bidi_count > 0 else "medium"`. The deduction already differs correctly (3 vs 2); the severity label just needs to match the documented intent.

## Low

### `--ascii` flag documented but not implemented
- **File**: `audit/cli.py:9`
- **Issue**: The module-level usage docstring lists `mcp-audit /path/to/mcp/package --ascii` as a valid invocation, but no `--ascii` argument is defined in the `argparse` setup. Running the command with `--ascii` raises "unrecognized arguments: --ascii".
- **Fix**: Either remove `--ascii` from the docstring, or implement the flag (presumably to strip non-ASCII output characters for terminals that can't render the emoji/em-dash in the markdown renderer).

### `open_calls` field on `ScanResult` is dead data
- **File**: `audit/ast_scan.py:234`
- **Issue**: `ScanResult.open_calls` is populated in `visit_Call` for every `open()` call, but it is not serialized by `scan_to_json` (all other fields are) and not consumed by `build_report`. The useful data (write vs. read classification) flows into `fs_write_calls` / `fs_read_calls` correctly, so this is not a correctness bug — but the field shows up in `ScanResult`'s public surface with no consumer, and any caller introspecting the dataclass directly will silently miss it from JSON output.
- **Fix**: Either serialize `open_calls` in `scan_to_json` alongside the other lists, or remove the field and rely solely on `fs_write_calls` / `fs_read_calls`.

### Module docstring says "8 audit dimensions" but 10 are implemented
- **File**: `audit/ast_scan.py:11`
- **Issue**: The file-level docstring lists eight numbered dimensions; the README and `findings.py` describe ten (dimensions 9 and 10 — string-content safety and unsafe deserialization — were added later). The docstring was not updated when those dimensions were added.
- **Fix**: Update the `ast_scan.py` module docstring to list all 10 dimensions to match the README and `findings.py`.
