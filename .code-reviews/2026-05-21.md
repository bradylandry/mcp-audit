## High

### Early `return` in `visit_Call` silences all subsequent dimension checks when `yaml.load(Loader=SafeLoader)` is encountered
- **File**: `audit/ast_scan.py:370`
- **Issue**: When `yaml.load` is matched and determined to have a safe `Loader=`, the visitor executes `self.generic_visit(node)` then `return`. This early return exits `visit_Call` entirely, preventing the rest of the method from checking the same call node for subprocess membership, HTTP client calls, filesystem writes, env reads, and inbound listeners. In practice this only matters if someone writes code that aliases `yaml.load` to a different call pattern that also happens to be a subprocess/HTTP call at the same AST node — unlikely but the logic structure is a latent trap. More concretely, `self.generic_visit(node)` is never called at the bottom of `visit_Call` for this path, which means child nodes of the safe `yaml.load` call (e.g., a URL or subprocess call nested in the argument expressions) are still walked (because `generic_visit` is called before `return`), but the flow is confusing and fragile: any refactor that moves the safe-loader guard into a helper could easily drop the `generic_visit` call, silently stopping child-node traversal.
- **Fix**: Replace the early return with a `flag_unsafe` boolean and fall through to the bottom `self.generic_visit(node)` that already exists. The safe-loader check should set `flag_unsafe = False` and `continue` past the append, not `return` from the entire visitor method. Alternatively, restructure so the safe-loader check is purely a guard on the `r.unsafe_deserialization_calls.append(...)` line, with no `return` at all.

## Medium

### Dead severity branch — `_check_string_content` always emits `severity="high"` regardless of `bidi_count`
- **File**: `audit/findings.py:338`
- **Issue**: The line `sev: Severity = "high" if bidi_count > 0 else "high"` assigns `"high"` unconditionally. The intent (visible in the comment above and the different `deduction` values on line 339) was to assign a lower severity for plain zero-width chars vs. bidi-override chars. As written, all zero-width-unicode findings are rated `high` but some get `deduction=2` and some get `deduction=3`, meaning the README's documented "−2 for plain, −3 for bidi" is implemented for the deduction but not reflected in the `severity` field consumers inspect (e.g., `--max-severity medium` will still fail on a plain zero-width-space finding, which was probably not the intent).
- **Fix**: Change the severity line to `sev: Severity = "high" if bidi_count > 0 else "medium"` to match the documented scoring methodology and make severity consistent with deduction magnitude.

### `--json` flag silently double-scans the target
- **File**: `audit/cli.py:189`
- **Issue**: The `--json` branch calls `scan_to_json(target, include_tests=args.include_tests)`, which internally calls `scan()` again. But `scan_result` was already computed by the earlier `scan(target, include_tests=args.include_tests)` on line 186. The second scan is redundant — for large packages it doubles the AST-walk time and (because it is a separate call) cannot share any cached state with the first. This is the only output mode that re-runs the scan; `--json-report` and the default markdown path both consume `scan_result` directly.
- **Fix**: Thread `scan_result` through: call `scan_to_json` after passing the already-computed result, or add a `scan_result_to_json(result)` variant to `ast_scan.py` that serializes an existing `ScanResult` rather than re-scanning.

### `_check_dependencies` only fires a finding when *all* deps are loosely pinned — partial loose pins are silently accepted
- **File**: `audit/findings.py:428`
- **Issue**: The condition `if loose and len(loose) == len(s.dependencies)` means the `MCP011` finding is only generated when every declared dependency lacks a `==` specifier. A package with 3 deps where 2 are pinned and 1 is loose (e.g., `requests>=2`) produces no finding at all. The README and scoring methodology say "loose version pins" trigger a low-severity finding, not "all deps are loose." This is a false-negative gap that could let a genuinely risky unpinned dep slip through.
- **Fix**: Change the condition to `if loose:` (remove the `len(loose) == len(s.dependencies)` guard). Adjust the finding title from "All N dep(s) use loose version pins" to "N of M dep(s) use loose version pins" to be accurate.

## Low

### `_dotted_name` is lossy for chained attribute access — documented limitation not surfaced to the user
- **File**: `audit/ast_scan.py:264`
- **Issue**: The docstring documents that `httpx.AsyncClient().get` resolves to `"AsyncClient.get"` (the leading `httpx.` is dropped because the intermediate `()` call breaks the chain). This means `httpx.AsyncClient().get("https://bad.example.com/")` is not in `_HTTP_CLIENT_FUNCS` (which has `"httpx.AsyncClient.get"` with the full prefix) and may be missed unless the instance-method heuristic in `_is_http_client_call` catches it via the "client" signal. The heuristic does catch this particular case, but the discrepancy between the catalog key format and what `_dotted_name` actually produces for chained calls is a latent source of future false negatives.
- **Fix**: Add a comment in `_HTTP_CLIENT_FUNCS` noting that chained-constructor patterns (`Cls().method`) are caught by the instance-method heuristic, not by exact catalog match, so both paths must be maintained consistently. Low priority — the current behaviour is correct; the risk is in future maintenance.

### `scan_to_json` serializes `suppressed_rules` silently as absent (not in `_call_dict`)
- **File**: `audit/ast_scan.py:718`
- **Issue**: `_call_dict` omits the `suppressed_rules` field from the `--json` raw output. A CI consumer that uses `--json` (raw scan) rather than `--json-report` to examine call sites cannot tell which findings were suppressed on a given line — the suppression information is lost. This is inconsistent with `--json-report`, which does include the `suppressions_applied` list.
- **Fix**: Either document that `--json` (raw scan) intentionally omits suppression data and users should use `--json-report` for suppression visibility, or add `"suppressed_rules": sorted(c.suppressed_rules)` to `_call_dict`. The latter is a one-line change.

### `_parse_deps` silently returns an empty list on Python 3.10 with a non-machine-readable note string
- **File**: `audit/ast_scan.py:580`
- **Issue**: On Python 3.10, `_parse_deps` returns `([], "pyproject.toml (3.11+ required for full parse)")`. The string `"3.11+ required for full parse"` is embedded inside `dep_source`, which is a prose field, not a structured flag. Downstream code in `_check_dependencies` and `render_markdown` just renders `dep_source` as-is, so the silent empty-deps on 3.10 causes the `MCP011` (loose-pin) check to never fire — the tool gives a false "no deps" result on a supported Python version. The pyproject.toml declares `requires-python = ">=3.10"` so 3.10 is a supported runtime.
- **Fix**: Either add `tomli` as a conditional dependency for Python < 3.11 (simple, adds one small dep), or parse the `[project.dependencies]` section with a minimal regex/line-scanner for 3.10 (slightly more work but keeps the zero-deps property). At minimum, emit a warning to stderr on 3.10 so users know dependency checks were skipped.
