## High

### Dead conditional in `_check_string_content` — both severity branches are `"high"`
- **File**: `audit/findings.py:338`
- **Issue**:
  ```python
  sev: Severity = "high" if bidi_count > 0 else "high"
  ```
  Both branches of the ternary assign `"high"`, so plain zero-width chars (ZWSP, zero-width non-joiner, etc.) receive the same severity label as bidi-override chars. The accompanying comment at line 335–337 explicitly says "bidi-override chars are higher severity" and "plain zero-width chars are usually a hidden-content attack vector but lower-severity than bidi inversion." The deduction still differs (3 vs 2), so scoring is partially correct, but the severity label surfaced in reports and in `--max-severity` CI gating is wrong for the non-bidi case. A repo with only ZWSP characters will fail a `--max-severity high` CI gate when the intent was to fail only on bidi-override.
- **Fix**:
  ```python
  sev: Severity = "high" if bidi_count > 0 else "medium"
  deduct = 3 if bidi_count > 0 else 2
  ```

## Medium

### Over-broad subprocess heuristic false-positives on `asyncio.run()` and similar
- **File**: `audit/ast_scan.py:375`
- **Issue**: The secondary subprocess detection branch at line 375 is:
  ```python
  any(name.endswith("." + sp.split(".")[-1]) for sp in _SUBPROCESS_FUNCS if sp.startswith("subprocess"))
  ```
  This matches any call whose dotted name ends with `.run`, `.call`, `.Popen`, `.check_output`, etc., regardless of the receiver object. Examples that would false-positive:
  - `asyncio.run(coro)` → ends in `.run`, matches `subprocess.run`
  - `runner.call(...)` → ends in `.call`, matches `subprocess.call`
  - `test_client.request(...)` → false negative (`.request` isn't in the suffix list), but the inconsistency is the point
  
  The primary `name in _SUBPROCESS_FUNCS` check on the preceding line already catches all the canonical forms. The heuristic suffix-match adds noise without adding meaningful coverage — actual subprocess calls use the module prefix (`subprocess.run`, `os.system`, etc.) and are already caught.
- **Fix**: Remove the secondary heuristic entirely. If instance-method detection is desired for subprocess-like objects (e.g. `popen_obj.communicate()`), add explicit named entries to `_SUBPROCESS_FUNCS` rather than using a broad suffix match:
  ```python
  if name in _SUBPROCESS_FUNCS:
      # ... (remove the `or any(...)` branch)
  ```

## Low

### Scoring methodology footer in `report.py` omits MCP012 and MCP013 rules
- **File**: `audit/report.py:116-117`
- **Issue**: The rendered markdown report includes a "Scoring methodology" section that lists severity levels and deduction ranges. This list was written by hand and does not include `ZERO_WIDTH_UNICODE` (MCP012, −2 to −3) or `INJECTION_PATTERN` (MCP013, −2) despite these being implemented rules with real score impact. A reader who audits a package that triggers MCP012 or MCP013 will see deductions that don't match the methodology description.
- **Fix**: Either generate the footer programmatically from the `RuleID` catalog + `findings.py` deduction constants, or add the missing rules to the hand-written list:
  ```
  - **High** (−2 to −3 each): subprocess with shell=True, dynamic exec (eval/exec),
    TLS verification disabled, zero-width/bidi-override unicode in string literals (MCP012),
    prompt-injection patterns in string literals (MCP013), unsafe deserialization (MCP004)
  ```

### `test_max_severity_low_fails_on_clean` is fragile — depends on clean fixture having loose deps
- **File**: `tests/test_audit.py:375-380`
- **Issue**: The test asserts that `--max-severity low` exits non-zero on the clean fixture, relying on the implicit assumption that `clean_mcp` always has at least one `low`-severity finding (the loose dep-pin finding). If `clean_mcp`'s dependencies are ever updated to use `==` pins (arguably the right thing to do for a test fixture), this test silently stops validating what it claims. The test will still pass — but now because of a different low finding or because of a bug — making the assertion misleading.
- **Fix**: Add an explicit stable low-severity signal to `tests/fixtures/clean_mcp` (e.g. two outbound hosts triggering MCP008 at low severity), or restructure the test to assert on a specific rule ID rather than the aggregate severity threshold.
