## High

### Dead severity branch makes bidi-override severity indistinguishable from zero-width severity
- **File**: `audit/findings.py:338`
- **Issue**: The conditional `sev: Severity = "high" if bidi_count > 0 else "high"` always evaluates to `"high"` regardless of `bidi_count`. The comment above distinguishes bidi-override (higher severity) from plain zero-width chars (lower severity), the deduction already varies (`3` vs `2`), but the `sev` variable is never "medium" or "low". The logic was clearly intended to differentiate severity between bidi and non-bidi findings but the else-branch was never filled in. This means a non-bidi zero-width char (e.g., ZWSP used as a style character) gets the same severity label as a Trojan Source bidi-override attack, making the report misleading and potentially causing CI gates to fire on false positives.
- **Fix**: Change line 338 to `sev: Severity = "high" if bidi_count > 0 else "medium"` (or the intended severity for non-bidi zero-width chars). Update `deduction` assignment on line 339 accordingly if needed. Ensure the README's scoring table and `render_markdown` methodology footer stay consistent.

## Medium

### `elif` in `visit_Call` skips HTTP-client detection when subprocess is matched first
- **File**: `audit/ast_scan.py:375-381`
- **Issue**: `visit_Call` dispatches via a chain of `if ŌĆ” elif ŌĆ” elif`. Subprocess detection is the first branch (`if name in _SUBPROCESS_FUNCS or ŌĆ”`). HTTP-client detection is in the next `elif`. If a name matches *both* ŌĆö which is not currently possible with the current catalogs, but future additions (e.g. a wrapper that happens to share a name) or the broad substring check `any(name.endswith("." + sp.split(".")[-1]) for sp in _SUBPROCESS_FUNCS if sp.startswith("subprocess"))` may produce an unexpected result ŌĆö the HTTP branch is silently skipped. Additionally, the `open()` bare-call branch is reached via `elif bare == "open"`, but a call named `open` that also matches `_FS_WRITE_FUNCS` (unlikely today) would be double-counted in `fs_write_calls`. The broader issue is that the single-level `elif` chain is the only guard; the scanner was built expecting catalogs to be mutually exclusive but this is not enforced or tested.
- **Fix**: Add a comment explicitly documenting the mutual-exclusivity assumption, or restructure into independent `if` blocks for dimensions that don't overlap (subprocess/exec, HTTP, filesystem, env, stdio, inbound) and enforce no overlap via an assertion or a unit test that verifies no name appears in more than one catalog.

### `--score-only` + `--max-severity` prints score before evaluating exit code ŌĆö misleading in CI
- **File**: `audit/cli.py:195-199`
- **Issue**: When `--score-only --max-severity high` is passed to a target that has high findings, `print(report.score)` emits the score to stdout before `_exit_code_from_max_severity` is called. The exit-code is returned correctly (1), but the printed score is `0` ŌĆö which can confuse CI log readers who see "0" printed and only notice the non-zero exit if they're checking `$?`. The `--json-report --max-severity` combination (lines 201-206) has the same pattern but is less confusing because the full JSON body makes the issue clear. Additionally, the `args.max_severity` attribute check in the `--score-only` branch uses `args.max_severity` (with underscore) while argparse stores it as the attribute name derived from `--max-severity`; this is fine because argparse converts hyphens to underscores, but it is worth documenting.
- **Fix**: In the `--score-only` + `--max-severity` branch, emit a human-readable exit-reason line to stderr (e.g. `"exit 1: findings at or above 'high' severity"`) so the printed score and the non-zero exit are reconciled for log readers.

### Dependency parsing silently falls back to empty list on Python 3.10
- **File**: `audit/ast_scan.py:578-581`
- **Issue**: On Python 3.10, `_parse_deps` returns `([], "pyproject.toml (3.11+ required for full parse)")` with no warning to the user. The `dep_source` string carries the explanation, but `_check_dependencies` in findings.py checks `if not s.dependencies: return` (line 419) and produces no finding. The user running on 3.10 silently gets no dependency-pin findings at all. The pyproject.toml for this project declares `requires-python = ">=3.10"`, meaning 3.10 is a fully supported configuration, yet dependency auditing is silently unavailable on it.
- **Fix**: Either add `tomli` as an optional dependency for Python 3.10 (`pip install tomli` is the standard backport), or emit a low-severity info finding when `dep_source` contains "3.11+ required" so the report makes the gap visible. Alternatively, implement a minimal TOML parser for the `[project.dependencies]` key only (the format is simple enough).

### `yaml.BaseLoader` treated as "safe" in deserialization check
- **File**: `audit/ast_scan.py:366-368`
- **Issue**: The set `safe_loaders` includes `"yaml.BaseLoader"` and `"BaseLoader"`. `yaml.BaseLoader` is not safe ŌĆö it avoids Python-object construction, which helps, but it does not have the same security guarantees as `yaml.SafeLoader` and is not the recommended safe option. Tagging it as safe could cause the scanner to skip a site that a human reviewer would flag.
- **Fix**: Remove `"yaml.BaseLoader"` and `"BaseLoader"` from `safe_loaders`, or add a comment explaining why it is treated as safe here and emit a lower-severity info finding instead of silently skipping it.
