## High

### Subprocess suffix heuristic produces false-positives on common patterns
- **File**: `audit/ast_scan.py:375`
- **Issue**: The subprocess detection uses a broad suffix heuristic: any dotted call whose final component is `run`, `call`, `check_call`, `check_output`, `Popen`, `getoutput`, or `getstatusoutput` is flagged as a subprocess, regardless of the receiver type. This catches `asyncio.run(main())`, `app.run()`, `task.run()`, `db.call("SELECT 1")`, `pool.Popen(...)`, and similar completely benign patterns. Each false-positive carries a `medium` severity finding with a −2 score deduction. An MCP server that uses `asyncio.run(main())` as its entrypoint will be incorrectly penalised 2 points even though the package has no shell exposure whatsoever. Confirmed with a live test: `asyncio.run(main())`, `app.run(host="0.0.0.0")`, `task.run()`, `pool.Popen(["x"])`, `db.call("SELECT 1")` all produce `subprocess_calls` entries.
- **Fix**: Restrict the suffix heuristic to receivers that actually plausibly reference `subprocess` — apply the same signal-list approach already used for HTTP client disambiguation (line 543): only match `*.run` / `*.call` / `*.Popen` when the receiver chain contains `subprocess`, `proc`, `process`, `shell`, `cmd`, or `command`. Alternatively, drop the suffix heuristic entirely and rely only on exact-name matches in `_SUBPROCESS_FUNCS`; the pattern `subprocess.run` is already in the set, and `os.system` / `pty.spawn` / etc. cover the remaining real-world cases.

## Medium

### Inbound-network dimension silently misses `app.run()` / `flask.Flask.run` due to `elif` chain ordering
- **File**: `audit/ast_scan.py:375`, `audit/ast_scan.py:471`
- **Issue**: `visit_Call` dispatches dimensions via a single `if / elif / elif ...` chain. The subprocess heuristic fires on any `*.run()` call — including `flask.Flask.run` and `uvicorn.run` — so those call sites are consumed by the subprocess branch before the `elif` for inbound-network (line 471) is ever evaluated. As a result, a package that starts a Flask server via `app.run()` gets one spurious `subprocess` finding (false positive) but zero `inbound_network` findings (false negative) — the actual risk is not surfaced at all. Confirmed: `app.run(host='0.0.0.0', port=8080)` produces `subprocess_calls=['app.run']` and `inbound_calls=[]`.
- **Fix**: Move inbound-network detection before the subprocess heuristic in the `if/elif` chain, or — preferably — check all applicable dimensions for each node (remove mutual exclusivity from the checks that should be independent). As a targeted fix, also check `name in _INBOUND_FUNCS` inside the subprocess branch and emit both findings when the name is an inbound pattern.

### Loose dependency pin check silently ignores partially-loose manifests
- **File**: `audit/findings.py:428`
- **Issue**: The condition `if loose and len(loose) == len(s.dependencies)` only fires when *every* declared dependency lacks an exact `==` pin. A package with nine loose deps and one pinned dep triggers no finding at all. In practice, mixed-pin manifests are the norm for packages that were retrofitted after a finding — an attacker shipping a malicious update to one loosely-pinned dep would not be flagged. The check also carries `deduction=0`, making it purely informational even when it does fire.
- **Fix**: Change the condition to `if loose:` (any loose dep triggers the finding). Adjust the title and detail to indicate how many of N deps are loosely pinned. Consider raising `deduction` to 1 when loose deps exist alongside exact pins (indicating partial, inconsistent pinning).

## Low

### Dead-code `else` branch in zero-width unicode severity assignment
- **File**: `audit/findings.py:338`
- **Issue**: The line reads `sev: Severity = "high" if bidi_count > 0 else "high"`. Both branches produce `"high"`, making the `else` branch unreachable dead code. The comment above (lines 335–337) explicitly states that plain zero-width chars (bidi_count == 0) "are usually a hidden-content attack vector but lower-severity than bidi inversion" — the intent was clearly `"medium"` for the non-bidi case, matching the lower `deduction=2` on line 339. The mismatch means all zero-width unicode findings, including plain ZWSP, are reported as `high` severity regardless of whether bidi-override chars are present.
- **Fix**: Change line 338 to `sev: Severity = "high" if bidi_count > 0 else "medium"`. The `deduction` differential (3 vs 2) is already correct.

### `--json` mode performs a redundant double scan
- **File**: `audit/cli.py:186–190`
- **Issue**: `main()` always calls `scan(target, ...)` at line 186 unconditionally. When `--json` is requested, it then calls `scan_to_json(target, ...)` at line 189, which internally calls `scan()` again — the target is walked and parsed twice. For large packages this doubles I/O and parse time. The first `scan_result` assigned at line 186 is discarded unused in the `--json` path.
- **Fix**: Restructure the `--json` branch to use the already-computed `scan_result`:
  ```python
  if args.json:
      sys.stdout.write(json.dumps(_scan_result_to_dict(scan_result), indent=2))
  ```
  where `_scan_result_to_dict` is the serialisation already implemented inline in `scan_to_json`. Alternatively, delay the initial `scan()` call until after the argument-mode check, only calling it for modes that need a `ScanResult`.
