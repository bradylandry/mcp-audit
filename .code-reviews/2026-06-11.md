## Medium

### Dead-code ternary: zero-width unicode severity always resolves to `"high"`
- **File**: `audit/findings.py:338`
- **Issue**: The severity assignment reads `sev: Severity = "high" if bidi_count > 0 else "high"`. Both branches return `"high"`, making the `bidi_count` condition dead code for the purpose of severity. The intent is almost certainly to assign a lower severity (e.g., `"medium"`) for plain zero-width chars vs. `"high"` for bidi-override, since the README and scoring docs distinguish them ("bidi-override chars are higher severity"). The `deduction` on line 339 correctly varies (`3 if bidi_count > 0 else 2`), but the `severity` field does not. This means the JSON report and `--max-severity` gating cannot distinguish zero-width-space from bidi-override by severity level.
- **Fix**: Change the else branch to the intended severity:
  ```python
  sev: Severity = "high" if bidi_count > 0 else "medium"
  ```
  This aligns with the README's stated hierarchy and makes `--max-severity medium` catch zero-width chars without triggering on bidi-overrides at the `high` threshold.

### Methodology footer misclassifies broad-credential env reads as high-severity
- **File**: `audit/report.py:117`
- **Issue**: The rendered markdown methodology section reads: `"- **High** (−2 to −3 each): subprocess with shell=True, dynamic exec, TLS verification disabled, suspicious env var reads"`. But `_check_env_vars` in `findings.py` classifies broad-credential env reads as `"medium"` severity with `deduction=1`. The footer overestimates their severity, misleading users who rely on the report to understand the scoring.
- **Fix**: Move "suspicious env var reads" (or more precisely "broad-credential env reads") to the medium-severity line in the footer:
  ```
  - **Medium** (−1 to −2 each): subprocess without shell, filesystem writes, inbound network listeners, dynamic URL destinations, broad-credential env var reads (AWS/GitHub/OpenAI tokens)
  ```
